If it gets stuck, then the cause for that is probably in the part of the <br>

<br>
I can see two other problems with this caching mechanism though:<br>
<br>
1. Items are never removed from the cache, so unless there is a limit to <br>
the number of different keys that can be used, the cache can grow <br>
indefinitely. You might want something like an LRU cache rather than a <br>
plain dictionary.<br>
<br>
<a href="https://docs.python.org/3/library/functools.html#functools.lru_cache" rel="noreferrer" target="_blank">https://docs.python.org/3/library/functools.html#functools.lru_cache</a><br>
<br>
2. If a lot of clients are requesting the same thing, you won't see any <br>
benefits from caching until the first request completes. So you could <br>
get a pattern like this:<br>
<br>
T=0: key A requested, A is not cached, start fetch #1 of A<br>
T=1: key A requested, A is not cached, start fetch #2 of A<br>
T=2: key A requested, A is not cached, start fetch #3 of A<br>
T=3: key A requested, A is not cached, start fetch #4 of A<br>
T=4: key A requested, A is not cached, start fetch #5 of A<br>
T=5: fetch #1 of A completes and is added to the cache<br>
T=6: key A requested, A is cached, return value immediately<br>
<br>
In this example, the value for A is fetched 5 times despite the caching  <br>
mechanism. If the fetching takes a long time compared to the rate at <br>
which requests are coming in, this effect gets worse at a quadratic <br>
rate: the total time spent fetching is the number of requests that come <br>
in during the fetching of the first request times the duration of the <br>
fetch.<br>
<br>
To avoid this, you could put a Deferred for the fetch operation in the <br>
cache or in a separate dictionary and if you get another request for the <br>
same key before the fetch completes, return that Deferred instead of <br>
starting another fetch.<br>
