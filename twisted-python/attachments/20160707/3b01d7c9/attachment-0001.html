<div dir="ltr"><div>I don't object to this specific change (we're on shiny new code), but want to offer some food-for-thought:</div><div><br></div><div>1) Is newer really better in cryptography?  Heartbleed affected 1.0.1, but not 1.0.0 and there are a bunch of vulnerabilities that only affect the newer libraries (<a href="https://www.openssl.org/news/vulnerabilities.html">https://www.openssl.org/news/vulnerabilities.html</a>).  It even makes sense that the older libraries have been more-thoroughly tested... so new code may just mean new vulnerabilities.</div><div><br></div><div>2) How does this impact regulated industries.  In healthcare (my current industry), changing a library (especially cryptography) could mean:</div><div><ul><li>An internal review to select a new version of the library</li><li>An internal change management process</li><li>Technical testing (perhaps a 3rd party audit)</li><li>A notification to clients of the change</li><li>Secondary reviews/testing at clients</li></ul><div class="gmail_extra"><div><div class="gmail_signature" data-smartmail="gmail_signature"><div>The intensity of this process depends on the risk level of the system and this could be a long and complicated process for some organizations.  Seems like a more deliberate deprecation policy would make it easier to plan ahead.</div><div><br></div><div>Clayton Daley</div><div><br></div></div></div>
