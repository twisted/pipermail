<br>Reasons include:<br>- the finish line gets moved: A reviewer says &quot;do this and then it&#39;ll be good to go into trunk&quot;.  The contributor does &quot;this&quot; and then another reviewer says &quot;oh just one more thing...&quot;.  Rinse.  Repeat.  This is hugely demotivational.  In many cases a better response from follow on reviewers would be to land the ticket, and then create another ticket for whatever other changes are needed.  This is not always possible but I think it should be done where it is possible.<br>
<br>- compatibility with some other unfinished ticket: The reviewer insists on compatibility with some other ticket, or waiting on some other ticket to land.  This is not necessarily a bad thing, but if the other ticket takes 6 months to land...well, can we blame people for wandering off and/or forgetting about things?  (to be fair, I think I&#39;ve only seen this once, and I can&#39;t recall what it was or find the ticket at the moment).<br>
<br>- I&#39;ve seen a lot of tickets that &quot;die&quot; at the &quot;this needs tests&quot; phase.  Requiring tests is a good thing, though.  I think the only way to help this problem is for reviewers to provide more guidance as to how exactly to create those tests.  Even figuring out where to put the tests is sometimes very difficult.  Exarkun has been pretty good about this in the last year or so, IIRC.<br>
<br>(It might be worth it to create a Trac keyword for this situation, maybe &quot;needs-tests&quot;.  This would make it easy to find those tickets, which might be a decent entry point for certain types of new contributors)<br>
<br>- Some people just seem to wander off.<br><br>- I can think of at least one ticket where the author stopped using Windows, and it was a Windows ticket.  <br><br>- That same Windows ticket is still open, though I tried to revive it.  It basically tries to do 2 things.  One of them is pretty nice (makes the StdIO protocol work on windows), but it&#39;s not even clear what the other one is trying to do.  So the ticket is in limbo until someone has the time to either figure it out or split the ticket in two.  <br>
<br>- I recall one ticket that got stalled, because noone was entirely sure who exactly had written the patch.  There was a third party who had contributed it to Twisted.  This was easily resolved with a google search and an email.  Of course the ticket is still open because it needs tests...<br>
<br>- One ticket was stalled because it required a &quot;merge forward&quot;.  Which is not really obvious how to do, especially since the DIvmod site died.<br><br>I&#39;m sure there are dozens of other reasons, but these are cases I can recall off the top of my head.<br>
