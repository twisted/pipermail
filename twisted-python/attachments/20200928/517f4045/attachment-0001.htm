<p>I guess this one is top-posting...  Anyways, I would personally default away from CI-provider-specific solutions.  It seems like a command line tool ought to suffice?  But maybe there's some security something or other I'm missing.  I just see that we presently use four CI systems so picking one and expecting to stick with it seems a bit naive.</p>
<p>Presumably someone will (helpfully) find an issue with this but I've copied this between a few projects.</p>
<p><a href="https://github.com/altendky/pyqt5-tools/blob/b0ecb9bf53ded53c28763d76f1e87d0b8180d2b8/twineontag.py">https://github.com/altendky/pyqt5-tools/blob/b0ecb9bf53ded53c28763d76f1e87d0b8180d2b8/twineontag.py</a></p>
<p>I know there are many other solutions.</p>
<p>Since this topic is a bit more general than just PyPI uploads I've also got a few comments about towncrier.  Per the link below I added towncrier to my RTD builds so that PRs will render the newsfragments.  Also, I started to vaguely think through some option for using towncrier that would allow for creating a release by pushing a tag rather than pulling a PR and doing various manual steps and modifying a generated news file etc.  No answers yet, just a thing I'd like to get going at some point.  Certainly `release by pushing a tag` is harder in bigger projects but I don't have in mind yet a reason it can't happen.</p>

<div>Maybe the release can be simplified and automated:</div>
<div> </div>
<div>* Build the source and binary wheel in GitHub actions as this can be done for Linux (Ubuntu), Windows and macOS.</div>
</div>
<div>* Host the download source and wheel files only on PyPi and publish them automatically from GitHub action on a new tag is created.</div>
<div>* Host the documentation only on the Read The Docs.</div>
<div>* Hos the API documentation on Read The Docs - might need some hacking, but at release we can create an API docs package as an artifact which is then pulled when Read The Docs documentation is created and copied as extra HTML files.</div>
<div>* Move Twisted blog to GitHub pages... or even read the docs with a separate theme like Crate [1] ... if the blog is still required.</div>
<div> </div>
<div>------</div>
<div> </div>
<div>With the above implemented the release should look like:</div>
<div> </div>
<div>* Each time the tests for a PR are executed, pydoctor will run and automatically create the API files as an artifact available for download.</div>
<div>* <strong>Manually</strong> create a new branch in which the version is updated and the changelog/news/release note is created and all the news fragment files are removed/ . Have the branch reviewed and approved with all the tests passed. When the pydoctor tests are executed, the API docs are created.</div>
<div>* The release branches can have a naming convention line 'release-20.0.0`. At first the release branch can have a release candidate version and a GitHub can automatically push the release to PyPi.</div>
<div>* <strong>Manually</strong> send an email to Twisted mailing list to announce the pre-release.</div>
<div>* Once the branch is merged, <strong>manually</strong> push a new tag</div>
<div>* The new tag should trigger the GitHub action for publishing the release on PyPi</div>
<div>* The new merge in master should trigger the Read The Docs build... which now will also include the static API docs pages.</div>
<div>* Once the release is done, <strong>manually</strong> send a new email to the mailing list.</div>
<div> </div>
<div>There are still many manual steps, but the only permissions required is commit to the repo.</div>
<div>This will no longer use any of the Twisted own infrastructure.</div>
