My application is an XMLRPC server, and an authenticated client should have rights to run some RPC methods, but not others.  Some methods will give access to limited data based on authentication.<br>

</blockquote><div><br>There's three parts to doing this:<br><ol><li>When setting up the portal, in addition to the credential checker that knows 
about regular users, also register a twisted.cred.checkers.AllowAnonymousAccess.</li><li>When you have no credentials, the xml-rpc layer should login to the portal using a twisted.cred.credentials.Anonymous credential. When you do have credentials, pass them in as you normally.</li>
<li>Now as part of a login your realm gets either a username, or a twisted.cred.checkers.ANONYMOUS as the avatar id that is being requested. Based on what it gets your realm should return a different business logic implementation (aka "avatar"); the anonymous one can do less, say.<br>
</li></ol>twisted.web.guard does #2; the way it is implemented, the business logic object (avatar) that is returned is a web Resource, so the realm can return a different Resource depending on whether there's HTTP credentials or not (the latter case being avatar id of ANONYMOUS). It's possible you just want to use guard, if you're relying on HTTP authentication - just return different XMLRPC objects. Alternatively, the XMLRPC layer could log-in to the portal, in which case it would get back different business logic objects with different capabilities.</div>
</div><br>An example of a full setup is twisted.protocols.ftp combined with twisted.tap.ftp, the ftp plugin for twistd, but the details are a bit spread out. You should be able to find all three of those parts though.<br>
