Since the Python 2 EOL date is rapidly approaching, I thought it was <br>
time we consider dropping Python 2 support.<br>
<br>
I personally find that Python 2 compat adds a huge amount of overhead <br>
when working on and maintaining Twisted, and think that with the current <br>
maintainer availability, dropping it sooner rather than later would have <br>
a beneficial effect on how much work we spend on shims/compat, <br>
complexity, and our ability to ship new features, as well as onboarding <br>
people who are interested in the project, but have no interest (or <br>
experience!) in Python 2.7.<br>
<br>
It is basically summed up by doing a feature freeze on an agreed-upon <br>
version of Twisted, that will be the last version released for 2.7. It <br>
would be abnormal in that it would get security fixes (our current <br>
policy is to only release them for current versions) and critical <br>
bugfixes, but would otherwise be frozen.<br>
<br>
One of my rationales is that from some analysis of PyPI download <br>
statistics, the vast majority of Python 2 users are using old versions <br>
of Twisted, while nearly all our Python 3 users are on the latest <br>
version. As such, I believe freezing a version that will get security <br>
updates but no new features would not be a massive loss to those stuck <br>
on Python 2 for whatever reason.<br>
<br>
Twisted's compatibility policy would still apply, ensuring that Python <br>
2/3 compatible software using Twisted would be able to use the older <br>
Twisted version on Python 2, and the newer version on Python 3, as you <br>
would usually expect.<br>
<br>
You can find the proposal here, in this handy-dandy Google Doc: <br>
<a href="https://docs.google.com/document/d/1S4CGgZC09blLIdk3Zo7wBa75A9_JuuH_3akkyjN0lik/edit" rel="noreferrer" target="_blank">https://docs.google.com/document/d/1S4CGgZC09blLIdk3Zo7wBa75A9_JuuH_3akkyjN0lik/edit</a><br>
