As a clarification to the above, parallelization of Python code across<br>
cores is not unique to Twisted; all Python code has this same<br>
limitation.<br>
<br>
To use multiple cores with Python code, you need multiple Python<br>
processes (as has been pointed out). One way to achieve this is to have<br>
the multiple processes talking to each other (using some kind of RPC<br>
protocol).<br>
<br>
Another way is to simply spawn some number of subprocesses (and Twisted<br>
has good support for running subprocesses). So, for example, if you<br>
write a CLI tool that can be told to run "part of your problem" then<br>
your parent Twisted process can simply spawn some number of those with<br>
appropriate arguments to split up the problem (e.g. give each process 1<br>
/ num_cores of the problem). This will incur some startup penalty as<br>
each process starts up (especially if you're using PyPy, which you<br>
should be if you care about speed) but is way simpler.<br>
<br>
Obviously, an RPC-style communication system avoids the startup penalty<br>
(but can be more complex).<br>
