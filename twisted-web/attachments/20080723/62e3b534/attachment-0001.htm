Since this is a common mis-conception (one I suffered from and have now disabused myself of) it&#39;s worth discussing.<br>
<br>
If my understanding is correct: twisted.cred uses the concept of an &quot;avatar&quot;. Avatars (I think...):<br>
<br>
&nbsp;* are protocol objects<br>
&nbsp;* represent the user<br>
<br>
In twisted.web, the Resource *is* the avatar. In twisted.mail.imap, the Mailbox is the avatar. In twisted.conch, the Shell is the avatar (and so on).<br>
<br>
I found this initially confusing, because in many web frameworks e.g. Zope, where I came from, the objects representing resources are:<br>
<br>
&nbsp;* long lived<br>
&nbsp;* the same instances serve &gt;1 HTTP request<br>
&nbsp;* instantiated at process start time<br>
<br>
It&#39;s also somewhat confusing because unlike other protocols, HTTP requests are very short-lived. It&#39;s worth noting you *can* have long-lived multi-instance t.w.Resource objects.<br>
<br>
In twisted, a common design pattern seems to be (I hope; I adopted it...)<br>
<br>
&nbsp;* have your realm return a root Resource i.e. one for &quot;/&quot;<br>
&nbsp;* attach your User object to that resource<br>
&nbsp;* have your root resource dynamically create the leaf resources via locateChild or childFactory, then attach the User object<br>
&nbsp;* when you need user info, get at it via e.g. self.user<br>
&nbsp;* let the leaf/root objects get GC-ed when the HTTP request is done<br>
<br>
I believe changes have been committed to twisted.web making this design pattern even more preferred:<br>
<br>
<a href="http://twistedmatrix.com/trac/changeset/23950" target="_blank">http://twistedmatrix.com/trac/changeset/23950</a><br>
<br>
Code which, by the way, I very much like the look of.<br>
<br>
